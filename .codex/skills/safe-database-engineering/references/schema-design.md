# Schema Design

## 목적

애플리케이션 객체가 아니라 데이터의 수명과 불변조건을 기준으로
관계형 스키마를 설계한다.

## 설계 순서

1. 데이터의 소유자와 변경 권한을 정한다.
2. 한 행이 표현하는 사실을 한 문장으로 적는다.
3. 식별자와 중복 규칙을 정한다.
4. NULL, 기본값, 범위, 상태 전이를 정의한다.
5. 관계와 삭제 동작을 정의한다.
6. 주요 쿼리와 쓰기 경합을 기록한다.
7. 제약조건과 인덱스를 설계한다.
8. 보존, 감사, 개인정보 요구사항을 반영한다.

## 식별자

- surrogate key와 natural key의 역할을 구분한다.
- 자연적으로 유일해야 하는 값은 별도 UNIQUE 제약을 검토한다.
- 외부에 노출되는 식별자와 내부 조인 키를 무조건 같게 만들지 않는다.
- 분산 생성 ID는 정렬성, 충돌, 인덱스 locality를 함께 검토한다.
- 복합키는 실제 불변조건일 때 사용하고 ORM 편의 때문에 피하지 않는다.

## 타입

- 금액은 부동소수점 대신 정확한 decimal 타입을 검토한다.
- 정밀도와 scale을 명시하고 반올림 규칙을 애플리케이션과 맞춘다.
- 시간 값은 의미를 구분한다.
  - 절대 시점
  - 지역 날짜와 시간
  - 날짜만 있는 값
- 시간대, DST, 직렬화 형식을 명시한다.
- 문자열 길이는 실제 계약과 인덱스 비용을 반영한다.
- enum은 값 추가, 이름 변경, 폐기 절차를 먼저 정한다.
- JSON은 유연성만 보고 선택하지 않는다.
  조회, 제약, 갱신 단위, 인덱싱 요구를 검토한다.

## NULL과 기본값

- `NULL`이 알 수 없음, 미적용, 미입력 중 무엇인지 정의한다.
- 빈 문자열, 0, 임의 날짜를 NULL 대용으로 사용하지 않는다.
- DB 기본값과 애플리케이션 기본값의 적용 시점을 구분한다.
- 새 NOT NULL 컬럼은 migration 단계와 기존 행 처리를 함께 설계한다.

## 제약조건

가능하면 다음 불변조건을 DB에서 보장한다.

- PRIMARY KEY
- UNIQUE
- FOREIGN KEY
- NOT NULL
- CHECK

애플리케이션의 사전 조회만으로 uniqueness를 보장하지 않는다.
동시 요청에서는 DB 제약조건과 오류 처리가 최종 방어선이다.

FK의 삭제와 갱신 동작은 명시적으로 결정한다.
`CASCADE`는 편리함이 아니라 데이터 수명 규칙으로 선택한다.

## 관계

- cardinality와 optionality를 실제 도메인 규칙으로 표현한다.
- 다대다는 연결 테이블의 속성과 수명을 검토한다.
- polymorphic relation은 FK 무결성 손실을 고려한다.
- soft delete는 모든 unique, FK, 조회, 보존 규칙에 영향을 준다.
- tenant key가 필요하면 PK, UNIQUE, FK, index에 일관되게 반영한다.

## 인덱스

인덱스는 예상이 아니라 접근 패턴을 지원해야 한다.

각 후보 인덱스에 다음을 기록한다.

- 지원하는 쿼리
- 필터, 조인, 정렬 조건
- 예상 선택도와 결과 행 수
- 읽기 개선과 쓰기 비용
- 기존 인덱스와의 중복 여부
- 생성 시 잠금과 디스크 영향

모든 FK 컬럼과 모든 필터 컬럼에 무조건 인덱스를 만들지 않는다.
실제 엔진과 workload를 확인한다.

## 상태와 이력

- 상태 값의 허용 목록과 전이를 정의한다.
- 현재 상태와 변경 이력의 요구를 구분한다.
- 생성, 수정, 삭제 시각의 의미와 작성 주체를 정한다.
- 감사 로그는 일반 수정 시각 컬럼을 대신하지 않는다.
- 변경 불가능한 기록은 수정과 삭제 권한을 별도로 제한한다.

## 개인정보와 보존

- 개인정보 여부와 민감도를 표시한다.
- 필요한 최소 데이터만 저장한다.
- 암호화, 마스킹, 토큰화의 책임 계층을 정한다.
- 보존 기간과 삭제 또는 익명화 절차를 설계한다.
- 운영 데이터가 비운영 환경으로 복제되는 경로를 확인한다.

## 검토 결과 형식

```md
## Entity or table

- Fact: 한 행이 표현하는 사실
- Owner: 데이터 소유자
- Key: 기본키와 자연키
- Invariants: UNIQUE, FK, CHECK, NOT NULL
- Lifecycle: 생성, 수정, 삭제, 보존
- Access patterns: 주요 읽기와 쓰기
- Indexes: 쿼리별 근거
- Sensitive data: 분류와 보호
- Open questions: 결정되지 않은 항목
```
