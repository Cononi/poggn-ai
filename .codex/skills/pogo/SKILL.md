# 구현 가이드 라인

## 1. 코딩하기 전에 생각하라.

- 가정을 거짓 없이 밝히고 확실하지 않다면 시작하지 말고 질문한다.
- 의도를 명확하게 이해하지 않았다면 여러 가지 의도를 제시한다.
- 의도와 다르게 더 좋은 방법론과 접근법이 있다면 제시하며, 필요하다면 반대 의견도 제시한다.
- 불문명하며, 명확하지 않은 사항을 제시한다면 멈추고 잘못된 지시에 대해서 지적이나 설명하고 질문한다.

## 2. 단순하게 넘기지 마라.

- 제시한 내용을 이미 알고 있다고 해서 문서를 대충 읽지 말아야 한다.
- 이렇게 처리하면 될껄이라는 판단 자체는 절대 금지다.
- 스킬같은 문서는 끝까지 제대로 읽어서 그대로 적용한다.

## 3. 복잡하게 생각하지 않는다.

- 문제를 해결하는 최소한의 코드만 작성하라. 그 이상은 작업하지 않는다.
- 지시한 내용에서 벗어나 다른 것을 읽어들여 그것까지 한번에 수정하려고 지시 내용에 본질을 흐리면 안된다.
- 한 번만 사용할 코드를 위해 추상화를 만들지 않는다.
- 지시하지 않은 "유연성"이나 "설정 가능성"을 추가하지 않는다.
- 발생할 수 없는 상황을 위한 예외 처리를 추가하지 않는다.
- 일어나지 않는 조건을 억지로 껴넣어 보기 힘들게 억지로 늘려서 과도한 코드를 만들지 않는다.
- 수백줄을 작성했는데 절반 이상으로 줄여서할 수 있다면 다시 작성한다.
- 과도한 코드라인이 생성됏는데 메소드나 클래스로 만들면 절반이상 줄일 수 있는 코드라면 그렇게 한다.
- 누군가 이 코드를 보고 최악이라고 말한다면 더 단순하고 유지보수와 신입 개발자가 발령받아 보고 판단할 수 있을 정도의 단순하게 만들어라.

## 4. GPU의 연산 체계처럼 움직여라.

- GPU의 강력한 연산 체계처럼 작동해야 한다.
- 분산 작업이 필요하다면 Multi Agent를 적극 활용한다.

### Multi Agent 사용 판단

Codex 공식 용어는 Subagents다. 사용자가 Multi Agent라고 말하면 Codex Subagents로 해석한다.
코드 수정, 버그 수정, 기능 개발, 리팩터링, 테스트 작성, QA, 보안/아키텍처 판단은 Subagents 사용을 기본 경로로 둔다. 메인 에이전트는 작업 단위와 파일 경계를 정하고 필요한 Subagents를 명시적으로 사용한다.

- 사용한다: 코드 수정, 버그 수정, 기능 개발, 코드베이스 탐색, 테스트/로그 분석, 리뷰, 여러 독립 작업의 병렬 구현.
- 사용하지 않는다: 문서 한두 줄 수정, 단순 질문 답변, 명령 출력 확인, 요구사항이 불명확한 작업, 동시 편집 충돌 가능성이 병렬 이점보다 큰 작업.
- 단, `.codex/state/pogo-settings.json`의 `subagent.auto`가 `true`이면 개발/수정/리뷰/QA 작업에서 Single Agent 예외를 쓰지 않는다.
- subagent auto가 켜져 있으면 메인 에이전트는 구현 전 최소 1개 이상의 관련 Subagent를 시작해야 하며, 분리 가능한 탐색/구현/검증은 병렬로 진행한다.
- hook은 Subagent 도구를 직접 실행하지 않는다. hook은 `$pogo-subagent-auto` shortcut을 `decision: block`으로 처리해 상태 변경/출력을 담당한다. `subagent.auto=true`이면 git commit/push/merge 전에 `.codex/state/subagent-evidence.json`의 PASS 증거를 요구한다.
- `$pogo-subagent-auto`, `$pogo-settings` 같은 hook shortcut이나 단순 상태 출력처럼 Subagent가 실질 작업 단위를 가질 수 없는 경우만 예외로 둔다.
- 에이전트 방식은 Single Agent와 Multi Agent 중 하나로 판단하되, 메인 에이전트는 추천 방식, 이유, 비용/위험, 진행 방식을 함께 제시한다.
- 모델은 공식 Codex 모델 ID인 `gpt-5.3-codex-spark`를 사용한다. 단, 이 모델을 사용할 수 없는 환경이면 중단하고 사용 가능 모델 확인을 요청한다.
- 모든 Subagent의 `model_reasoning_effort`는 기본 `high`로 둔다. 워커, 검증, 보안, 아키텍처, 버그 수정, 리팩터, 테스터 agent는 extra-high 값인 `xhigh`를 사용하고 developer instructions에 extra-high scrutiny profile을 적용한다.
- Subagents는 현재 sandbox와 approval 정책을 상속한다는 전제를 둔다.
- 병렬 작성이 필요한 경우에도 최종 병합과 충돌 판단은 메인 에이전트가 한다.

### 계획 수립 판단

작업 시작 전에 pogo 계획과 evidence-driven-sdd 계획 중 무엇이 더 적합한지 판단한다.

- pogo 계획을 쓴다: T0 수정, 좁은 버그 수정, 리팩터링 없는 단순 구현, 외부 동작 변화가 없거나 작고 명확한 변경.
- evidence-driven-sdd 계획을 쓴다: 사용자 동작, API, 데이터 모델, 비즈니스 규칙, 보안, 결제, 개인정보, 마이그레이션, 동시성, 호환성에 영향이 있는 T1/T2 변경.
- 둘 다 애매하면 evidence-driven-sdd 기준으로 등급을 먼저 정하고, 불필요한 문서화가 되지 않도록 최소 Spec만 작성한다.
- 계획 비교가 필요한 작업에서는 `pogo-planner` Subagent에게 pogo 방식과 evidence-driven-sdd 방식의 장단점, 위험, 검증 기준을 비교하게 하고 메인 에이전트가 최종 선택한다.

### 전문 스킬 라우팅

작업 영역에 맞는 전문 스킬을 메인 에이전트가 먼저 판단한다. Single Agent로 진행하면 메인 에이전트가 해당 스킬을 직접 적용하고, Multi Agent로 진행하면 같은 관점의 전문 Subagent에게 위임한다.

- `architecture`: 여러 모듈, 경계, 의존성, public contract, 장기 구조 판단이 필요한 작업.
- `backend`: API, 서버 도메인 로직, 인증/권한, 데이터 저장, job/queue/integration 작업.
- `front`: 컴포넌트, 화면 상태, form, routing, client API 연동, 접근성/반응형 구현 작업.
- `designer`: 사용자 흐름, 정보 구조, 레이아웃, 시각 계층, UX 상태, 디자인 일관성 판단 작업.
- `security`: 인증, 권한, 입력 검증, 민감 데이터, secret, dependency, 네트워크, 파일, abuse case 검토가 필요한 작업.
- 여러 영역이 걸리면 architecture로 경계를 먼저 정하고 backend/front/designer/security를 병렬 또는 순차로 배치한다.
- 전문 Subagent 결과는 원시 로그가 아니라 결정, 변경 범위, 검증 증거, 남은 위험으로 요약해 메인 에이전트가 병합한다.

### Subagents 운영 방식

재사용 가능한 커스텀 에이전트는 `.codex/agents/`에 둔다.

1. 메인 에이전트가 작업을 독립 단위로 나누고, 코드 변경 계열이면 Subagents 사용을 먼저 시도한다.
2. 버그 수정은 `pogo-bug-agent`가 재현, 원인 축소, 수정 방향, 회귀 검증을 우선 담당한다.
3. 구현 Subagent는 승인된 계획과 범위 안에서만 작업하고 결과 요약, 변경 파일, 검증 결과, 남은 위험을 반환한다.
4. 구현이 끝난 작업 중 보안 영향이 있으면 먼저 `pogo-security-agent`에게 보안 검토를 의뢰한다.
5. 구현과 필요한 보안 검토가 끝나면 `pogo-verifier`에게 완료 작업 요약, 변경 파일, 검증 증거, 남은 위험 확인을 맡긴다.
6. 메인 에이전트는 작업지시와 결과의 유사도/완성도를 0-100으로 판단한다. 95 미만이면 이유와 최소 재작업 범위를 지정해 다시 지시한다.
7. 95 이상이고 리팩터링 필요성이 확인된 경우에만 `pogo-refactorer`에게 최소 파일 목록, 리팩터링 목적, 보존해야 할 동작을 전달한다.
8. 최종 단계는 `pogo-tester`에게 넘긴다. 메인은 기능 요약, 변경 파일, 테스트 초점, 관련 테스트 경로만 간결하게 전달한다.
9. 테스트 코드 작성과 테스트 실행은 `pogo-tester`가 수행한다.
10. Subagents의 원시 로그를 메인 흐름에 그대로 붙이지 말고, 필요한 근거와 결론만 요약한다.
11. `subagent.auto=true`에서 작업을 완료하려면 `.codex/state/subagent-evidence.json`에 현재 branch, HEAD, 변경 파일 목록, `pogo-verifier` 또는 `pogo-tester`의 `PASS` 결과를 남긴다. evidence는 현재 git 상태와 일치해야 하며, 24시간 이상이면 stale로 보고 삭제 후 다시 생성한다.

## 5. 기존 코드를 수정할 때

- 반드시 필요한 부분만 명확히 찾아서 수정한다.
- 자신이 만든 문제에 관련 코드만 일관되게 처리하라.
- 마음대로 주변 코드, 주석, 포맷을 개선에 시도하지 마라.
- 본인이라면 다르게 작성했더라도 기존 스타일을 그대로 따라라.
- 관련 없는 죽은 코드(dead code)를 발견했다면 언급만 하고 삭제하지 마라.

## 6. 당신의 변경으로 고아(orphan)가 된 것들이 생겼다면

- 당신의 변경 때문에 사용되지 않게 된 import, 변수, 함수, 클래스, 파일은 제거하라.
- 고아가 되어 필요가 없는 코드를 가지고 지속적으로 활용하려해서 토큰 낭비 하지말고 제거하라.
- 테스트 기준은 변경된 모든 줄은 사용자의 요청과 직접적으로 연결될 수 있어야 한다.

## 7. 목표 중심으로 실행하라

성공 기준을 정의한다. 검증될 때까지 반복한다.
작업을 검증 가능한 목표로 변환한다:

- "유효성 검사 추가" → "잘못된 입력에 대한 테스트 작성 후 통과 확인"
- "버그 수정" → "버그를 재현하는 테스트 작성 후 통과 확인"
- "X 리팩토링" → "리팩토링 전후의 테스트 통과 확인"

다단계 작업의 경우 간략한 계획을 수립한다:

1. [단계] → 검증: [확인 사항]
2. [단계] → 검증: [확인 사항]
3. [단계] → 검증: [확인 사항]
성공 기준이 명확해야 독립적인 작업이 가능하다. "작동하게 만들기"와 같은 모호한 기준은 불필요한 재질의를 야기한다.

지침 작동 확인: Diff 내 불필요한 변경 감소, 복잡성으로 인한 재작성 빈도 감소, 구현 전 질문을 통한 명확한 의사결정 증대.
